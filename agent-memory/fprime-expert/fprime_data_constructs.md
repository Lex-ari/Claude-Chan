# F Prime Data Constructs: Commands, Events, Channels, and Parameters

## Commands

Commands enable user interaction with components. Unlike ports (component-to-component), commands are designed for ground/user control.

### Command Properties

1. **opcode**: Numerical ID (unique per component, auto-offset by base ID)
2. **mnemonic**: Text name (prepended with component instance name for system uniqueness)
3. **arguments**: Primitive or F Prime data types supplied from ground
4. **synchronization kind**: Controls execution context
   - `sync`: Runs on dispatcher thread
   - `async`: Runs on component thread with priority
   - `guarded`: Runs on dispatcher thread with mutex protection

### Command Flow

1. Ground sends command to `Svc::CmdDispatcher`
2. Dispatcher extracts opcode, looks up handling component
3. Dispatcher invokes command handler on target component
4. Component processes command, calls `cmdResponse_out()` with status
5. Dispatcher receives response and can forward to sequencer if applicable

**Critical Pattern**: Command handler MUST call `cmdResponse_out(opCode, cmdSeq, Fw::CommandResponse)` when complete.

### Command Dispatcher Port Connections

Each commanding component needs three parallel port connections to dispatcher:
- **Command registration port** (`CmdReg`): Registers opcodes during init
- **Command dispatch port** (`CmdDisp`): Receives commands to execute
- **Command response port** (`CmdStatus`): Reports completion status

## Events

Events represent a log of system activities - the execution history of the embedded system.

### Event Properties

1. **id**: Numeric ID (offset by component base ID)
2. **name**: Text identifier (prepended with component instance name)
3. **severity**: Defines event importance
   - `DIAGNOSTIC`: Debug messages, usually not downlinked
   - `ACTIVITY_LO`: Background task info
   - `ACTIVITY_HI`: Foreground task info
   - `WARNING_LO`: Low-severity warnings
   - `WARNING_HI`: High-severity warnings (system still functions)
   - `FATAL`: System must reboot
   - `COMMAND`: Command execution tracing
4. **arguments**: Variable data for the event
5. **format string**: C-style format string for text reconstruction

### Event Flow

1. Component calls autogenerated `log_<severity>_<event_name>()` function
2. Base class retrieves time tag from time source
3. Event sent to two independent output ports:
   - **Binary log port**: For downlink to ground (`Svc::EventManager`)
   - **Text log port**: For onboard console (`Svc::PassiveConsoleTextLogger`)
4. `Svc::EventManager` processes events, responds to FATAL events

**Why:** Events capture all activities, sporadic but all captured for downlink.

## Channels (Telemetry)

Channels represent current system state snapshots. Unlike events (historical), channels are the "now" of the system.

### Channel Properties

1. **id**: Unique channel ID (offset by component base ID)
2. **name**: Text name (prepended with component instance name)
3. **data_type**: Type of channel value (primitive or complex)
4. **update mode**: 
   - `on_change`: Only downlink when value changes
   - Default: Always downlink on every update

### Channel Flow

1. Component calls `tlmWrite_<channel_name>(value)` anytime state changes
2. Base class retrieves time tag internally
3. Value written to `Svc::TlmChan` (telemetry database)
4. `Svc::TlmChan` uses double-buffered storage
5. Rate group periodically calls TlmChan to send current snapshot to ground

**Why:** Channels are sampled at set rate, not all values permanently stored - just current state.

## Parameters

Parameters are non-volatile stored values that influence software behavior without requiring code updates.

### Parameter Properties

1. **id**: Unique ID (offset by component base ID)
2. **name**: Text name (prepended with component instance name)
3. **data_type**: Type of stored value (primitive or complex)
4. **default value**: Used if parameter cannot be retrieved

### Parameter Flow

**During Initialization:**
1. `Svc::PrmDb` loads parameter file from filesystem
2. Public `loadParameters()` called on each component
3. Component retrieves parameters via `getPrm` port
4. Parameters stored in component base class

**During Runtime:**
1. Component calls `paramGet_<parameter_name>(Fw::ParamValid& valid)` to retrieve
2. Check `valid` status:
   - `Fw::PARAM_UNINIT`: loadParameters() never called
   - `Fw::PARAM_VALID`: Successfully retrieved
   - `Fw::PARAM_INVALID`: Retrieval failed, no default
   - `Fw::PARAM_DEFAULT`: Retrieval failed, default provided

**Parameter Updates:**
- Parameters can be updated by command
- `parameterUpdated(FwPrmIdType id)` virtual method called on update (override to handle notifications)
- Updated values saved to `Svc::PrmDb`
- PrmDb can be commanded to persist to filesystem

### External Parameters

For parameters managed outside component:
- Implement `ParamExternalDelegate` interface
- Register via `registerExternalParameters()`
- Implement `deserializeParam()` and `serializeParam()` methods
- Useful for decoupled parameter storage

## Serialized Ports Note

`Svc` components use serialized ports to generically handle different data types for uplink/downlink. This allows command dispatcher, event manager, and telemetry database to work without knowing specific types being passed through.

**How to apply:** When designing components that interact with commands/events/telemetry, understand that autocoded ports handle serialization automatically. Manual serialization only needed for custom port types or when crossing address-space boundaries (Hub pattern).
